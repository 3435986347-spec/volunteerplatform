# 恒德志愿者平台 · 生产部署说明（裸 jar + systemd + nginx）

适用部署形态（已定）：**方案C 分离部署**——同一台服务器上，nginx 托管**后台管理前端**静态文件并把 `/api` 反代给 Spring Boot 后端；后端以**裸 jar + systemd** 方式常驻。小程序端只依赖后端 HTTPS 域名，不涉及静态托管。

配套制品（仓库 `部署/` 目录）：

| 文件 | 用途 | 服务器目标位置 |
|---|---|---|
| `nginx.conf` | 前端静态 + `/api` 反代 + 安全头 | `/etc/nginx/conf.d/hengde-admin.conf` |
| `hengde-api.service` | 后端 systemd 服务单元 | `/etc/systemd/system/hengde-api.service` |
| `api.env.example` | 生产环境变量模板 | 填真值后 → `/etc/hengde/api.env`（600） |

> 本文只讲**服务器上的安装动作**。域名备案、微信小程序、短信/OSS 等第三方开通与交付项见同目录 `上线checklist.md`；本地联调见 `运行说明.md`。

---

## 0. 服务器前置

- Linux 服务器（建议 2C4G 起），安装：**JDK 17**、**nginx**、**MySQL 8.x**、**Redis 5+**（中间件要求与安全注意见 `上线checklist.md` 第 2 节；MySQL/Redis 端口不对公网开放）。
- 建空库：`CREATE DATABASE hengde_volunteer DEFAULT CHARACTER SET utf8mb4;`（只建库；表由 Flyway 首启自动建到 V19 并写种子数据）。
- 目录与用户约定（下文按此写，换路径需同步改 service/nginx 配置）：

```bash
sudo useradd -r -s /usr/sbin/nologin hengde          # 专用运行用户
sudo mkdir -p /opt/hengde/api/config                 # jar + application-prod.yaml
sudo mkdir -p /opt/hengde/admin-web                  # 后台前端静态
```

## 1. 构建制品（开发机）

后端 jar（父 POM 无聚合段，须按依赖顺序逐模块 install；Windows 用 `mvnw.cmd`）：

```bash
cd 代码/hengde-volunteer-parent
./mvnw install -N
for m in common auth organization activity publicity user data api; do
  ./mvnw clean install -DskipTests -f ../hengde-volunteer-$m/pom.xml
done
```

产物：`代码/hengde-volunteer-api/target/hengde-volunteer-api-1.1-SNAPSHOT.jar`。

前端无构建步骤，产物就是 `volunteer-platform-back/` 下的 **`index.html` + `assets/`** 两样（README/联调指南/uploads 参考资料**不要**上服务器；nginx 配置对它们也有兜底 deny）。

---

## 2. 后端安装（systemd）

```bash
# 1) jar 就位
sudo cp hengde-volunteer-api-1.1-SNAPSHOT.jar /opt/hengde/api/

# 2) 环境变量：从模板复制并填真值（哪些必填、密钥怎么生成见模板内注释）
sudo install -D -m 600 api.env.example /etc/hengde/api.env
sudo vi /etc/hengde/api.env

# 3) 协议正文等长文本配置（环境变量放不下的），写 prod 覆盖配置：
sudo vi /opt/hengde/api/config/application-prod.yaml
```

`application-prod.yaml` 最小示例（该文件含正式文本，已被 `.gitignore` 排除，只存在于服务器）：

```yaml
hengde:
  auth:
    agreement-text: |
      雷州市恒德爱心公益协会志愿者服务协议
      第一条 ……（协会提供的正式协议全文）
```

```bash
# 4) 服务单元就位并启动
sudo cp hengde-api.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now hengde-api

# 5) 看首启日志（Flyway 建表 + 「生产配置守卫校验通过」+ Tomcat started）
sudo journalctl -u hengde-api -f
```

**首启失败最常见原因就是守卫 fail-fast**（属预期设计，宁可起不来不让弱配置上线）。报错会逐条列出问题，对照补 `/etc/hengde/api.env`：profile 未设 prod / AES·HMAC 密钥为空或 dev 默认 / 超管密码仍 admin123 / dev-login 未关 / 协议版本空或正文仍占位。改完 `systemctl restart hengde-api`。

验证后端在线：`curl -i http://127.0.0.1:8080/api/a/auth/login -X POST`（返回 JSON 错误体即在线；此时还未配 nginx，直连 8080）。

## 3. 前端安装（nginx）

```bash
# 1) 静态文件就位（只拷这两样）
sudo cp -r index.html assets /opt/hengde/admin-web/

# 2) （可选）启用高德地图选点：编辑 index.html 填 __AMAP_KEY__（及安全密钥或代理，见文件内注释）
sudo vi /opt/hengde/admin-web/index.html

# 3) nginx 配置就位：改 server_name 为真实域名/IP 后校验生效
sudo cp nginx.conf /etc/nginx/conf.d/hengde-admin.conf
sudo vi /etc/nginx/conf.d/hengde-admin.conf
sudo nginx -t && sudo nginx -s reload
```

浏览器打开 `http://<域名或IP>/` 应见后台登录页；前端 `api.js` 在 80/443 端口自动用同源 `/api`，无需任何前端改动。

> 发版更新前端：直接覆盖 `/opt/hengde/admin-web/` 下文件即可——nginx 已配 `no-cache + ETag` 协商缓存，浏览器下次请求即拿到新版，无需让用户清缓存。

## 4. HTTPS（有备案域名后）

```bash
sudo dnf install certbot python3-certbot-nginx     # Debian/Ubuntu: apt install
sudo certbot --nginx -d <域名>                      # 自动改写 hengde-admin.conf：加 443 块 + 80 跳转
sudo systemctl enable --now certbot-renew.timer    # 自动续期
```

微信小程序「request 合法域名」必须是 **HTTPS + 已备案**（流程见 `上线checklist.md` 第 1/3 节）；后台前端对 HTTPS 零改动。

---

## 5. 上线冒烟（后台侧）

完整冒烟清单见 `上线checklist.md` 第 6 节；后台/部署侧最少验证：

1. `https://<域名>/` 打开登录页，**控制台无报错**；
2. 超管登录成功 → 概览页**数据看板有数字**、待办卡按权限显示；
3. 发布一个活动、传一张轮播图——确认返回**真实 OSS URL** 且图片可访问（上传链路 nginx 16m→multipart 12MB→业务 10MB 已对齐）；
4. `https://<域名>/api/doc.html` 应为 **403**（生产已屏蔽接口文档，预期行为；联调需要时临时注释 nginx 对应 location）；
5. `https://<域名>/README.md`、`/uploads/` 任意路径应为 403。

---

## 6. 升级与回滚

- **后端发版**：替换 `/opt/hengde/api/` 下 jar → `sudo systemctl restart hengde-api`。Flyway 自动跑新增迁移；**Flyway 不会自动回滚**，升级前先备份库：
  `mysqldump -u<user> -p hengde_volunteer | gzip > hengde_$(date +%F).sql.gz`
- **前端发版**：覆盖 `/opt/hengde/admin-web/`，即时生效（见第 3 节）。
- **回滚**：后端换回旧 jar 重启（若新版本带了迁移则需先恢复备份库）；前端覆盖回旧文件。

## 7. 多组织（白标）

每个社会组织 = 独立库 + 独立 `/etc/hengde/<org>.env` + 独立域名/小程序，跑同一份 jar：复制一套 service 单元（改名、改 env 路径、改端口）+ 一个 nginx server 块即可，互不共享数据。详见 `上线checklist.md` 第 7 节。
