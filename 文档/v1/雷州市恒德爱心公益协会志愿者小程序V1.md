###  恒德志愿者平台 V1 — 两人分工方案

 仓库当前是脚手架：只有父 POM 存在，子模块/业务代码/迁移脚本都还没建。V1 范围（见
 雷州市恒德爱心公益协会志愿者小程序V1.md）覆盖四块后端领域 + 一套小程序前端 + 一套后台管理端。

 约束（已与你确认）：
 - 前端范围 = 小程序(首页/社区/我的) + 后台管理端，负担很重。
 - 两人后端能力基本对等。

 由此定下两条原则：
 1. 地基放在不画前端的那个人手上（关键路径，先搭好解阻塞）。
 2. 画前端的人后端切薄一点，拿耦合度低、偏 CRUD 的领域，避免一人扛前端+重后端。

---
 开发者 A — 纯后端主力（地基 + 认证 + 权限 + 活动）

 A 不碰前端，因此承担更重的后端与关键路径。

 阶段 0 · 地基（最优先，做完才解两人阻塞）
 - 父 POM <modules> 声明子模块；按 CLAUDE.md 扁平结构，各模块 parent 直指 hengde-volunteer-parent。
 - hengde-volunteer-common：统一返回体 / 全局异常 / 错误码、Sa-Token + JWT + Redis 鉴权配置、OSS(或 MinIO)
 上传封装、火山短信通道、Flyway 基础、Testcontainers-MySQL 测试基座(test-jar 供各模块 @Import)。
 - hengde-volunteer-api：唯一 @SpringBootApplication 启动类、全局配置、resources/db/migration/
 迁移目录、Knife4j/OpenAPI。
 - 依赖注入按约定走 setter 注入；测试统一 @SpringBootTest + Testcontainers。

 auth 领域（V1 角色：游客/志愿者/管理团队）
 - 小程序登录、志愿者注册（身份证二要素实名、短信验证码、企业微信群前置校验、未成年紧急联系人、协议手写签名、会员费门槛预留）。
 - 后台账号登录、忘记/修改密码、超管账号初始化预置、多角色登录态与鉴权。

 organization·子账号与权限（RBAC） — 与鉴权强耦合，归 A
 - 管理团队独立子账号；菜单权限 / 数据权限 /
 审核流程权限分离；最高权限账号统一分配；界面水印（手机尾号+姓名+部门）。

 activity 领域
 - 活动发布：子项目时间段(多段)、积分倍率(负责人×1.4/管理团队×1.2)、报名限制(全平台/指定分队)、报名审核开关、报名须
 知弹窗、复制活动、唯一递增编号、活动补录。
 - 活动报名：志愿者自主报名(选时间段)、同小组代报名、全平台同时间段防重复报名。
 - 活动管理：报名审核、评论/总结、报名数据导出、修改/删除。

 开发者 B — 后端较轻 + 全部前端（小程序 + 后台管理端）

 B 扛全部前端，后端只拿耦合低、偏 CRUD 的领域。

 organization·业务部分
 - 志愿小组：发起/自主命名、管理员(≤3)、审批加入、移除成员、一人一组、按名称/编号搜索、同组代报名；后台搜索/批量导入
 /解散/组建/组长转移/变更记录/历史小组数据、新建小组后台审核。
 - 归属分队：类型后台自定义(学校/乡镇)、每类型加入上限、未归属看负责人信息、已归属看同队名单、后台设置同队可见信息范
 围、申请审核权归分队负责人(总账号看全量记录)。

 publicity 领域
 - 轮播图：后台预设尺寸+上传裁剪+跳转推文/小程序。
 - 公告栏：插图+跳转+预设裁剪尺寸。
 - 全局搜索：首页搜索框，common/api 聚合层（依赖各模块提供检索接口，见协作缝）。

 前端（重头）
 - 小程序三 Tab：首页(轮播/公告/搜索/活动入口)、社区(V1 占位)、我的(注册/登录/实名/我的小组/我的分队/我的活动)。
 - 后台管理端：志愿者管理(搜索+下拉筛选/批量导出/详情/重置密码/改部门)、活动管理、小组与分队管理、子账号与权限配置界
 面、轮播/公告管理。

---
 协作缝（必须两人共同约定的接口）

 - 统一返回体 + 错误码：A 在 common 定义，B 前端按此对接。
 - API 契约先行：A 先产出 auth + activity 的 OpenAPI(Knife4j)，B 前端用 mock 数据先画静态页，接口上线后切真接口。
 - 权限模型：A 实现后端 RBAC，B 实现后台权限配置界面 —— 菜单/数据/审核三类权限的结构要一起定。
 - 小组代报名 ↔ activity：B 的小组模块调用 A 的 activity 报名 service，约定 service 接口，避免循环依赖。
 - 全局搜索聚合：B owns 聚合，但各领域(A 的 auth/activity、B 的
 organization/publicity)需各自提供检索入口，统一聚合协议。
 - 文件上传：A 在 common 封装 OSS，B 前端统一调用。

 排期建议（粗粒度）

 1. 第 1 阶段：A 搭地基(0 阶段)；同时两人敲定返回体+API 契约+权限模型。B 先搭前端工程骨架、画静态页(mock)。
 2. 第 2 阶段：A 做 auth+RBAC，B 做小组/分队 后端 + 对应前端；auth 接口就绪后 B 接小程序注册/登录真接口。
 3. 第 3 阶段：A 做 activity，B 做 publicity + 全局搜索 + 剩余后台页面；联调代报名、搜索聚合。

 验证方式

 - 后端：各领域模块放测试用 @SpringBootApplication + @SpringBootTest + Testcontainers-MySQL（需本机 Docker），./mvnw
    test -pl <module>。
 - 联调：api 模块 ./mvnw clean install -DskipTests 起单体，Knife4j 看接口；B 前端连本地 api
 跑通注册→登录→报名→公示主链路。
