**发现 / 已就地修正：**

1. **（清单数据修正）第 5 步示例身份证号 `440000200001011234` 校验不过** —— dev 下实名校验虽放行第三方，但 `RealNameServiceImpl` 仍先做 `IdcardUtil.isValidCard` 格式+校验位校验，假号会报「实名认证失败」。**本清单已改用合法号 `440882200001011235`**（雷州市段，校验位正确）。这是清单示例数据问题，非后端缺陷。
2. **（真缺陷，已修复）志愿小组「待审核加入申请」无列表接口** —— 原本组长批准/拒绝加入要传 `memberId`，但 `GET /v/organization/groups/{id}/members` 只返回 ACTIVE 成员、`/a` 侧 `/applications` 只列**建组**申请，没有任何接口列出某组**待审核的加入申请**及其 `memberId`，加入审批链在 API 层断了。**已补 `GET /v/organization/groups/{id}/join-applications`**（仅组长/管理员可见，返回 PENDING 加入申请及 `memberId`/申请人/申请时间，排除建组时的 PENDING 组长行），前端凭返回的 `memberId` 即可直接调 approve/reject。已加 3 条单测覆盖（列出待审且回 memberId / 非成员拒 / 普通成员拒）。
3. **（清单提示）第 6 步时间窗活动** —— `enrollDeadline` 不填默认 `startTime − 24h`，且建活动校验 `enrollDeadline ≤ startTime`。要让「此刻」能报名又能签到，需**显式传 `enrollDeadline = startTime`，并把 `startTime` 设为当前时刻稍后**（签到只看「开始前2h~结束后2h」窗口，不要求已到开始时刻；签退只看「结束后2h 内」，均可手动 start/finish 触发）。

## 第 6 步 · 活动闭环进阶（签到→签退→时长→秘书确认→积分，V1.1）

> ⚠️ **时间窗口敏感**：GPS 签到要求在「活动开始前 2h ~ 结束后 2h」窗口内、且负责人已点「活动开始」。
> 想顺畅测，**另发一个活动把 `startTime` 设为当前时间稍后、`endTime` 设为再晚一两小时**，并填 `lat/lng`。
> 关键：**必须显式传 `enrollDeadline = startTime`**（不填会默认成 `startTime − 24h`，且建活动校验 `enrollDeadline ≤ startTime`），否则「此刻」会报「报名已截止」报不上名。签到只看时间窗、不要求已到开始时刻；`needAudit:0` 可让报名直接通过省去审核。

## 第 8 步 · 鉴权 / 负向用例（顺手验证）

- [x] **不带 token** 调任意 `/v/**` 或 `/a/**`（非白名单）→ 预期 **401 未登录**。
- [x] **token 串用**：拿 `/v` token 调 `/a/**`（或反之）→ 预期 401/403（两端登录态隔离）。
- [x] **子账号越权**：用没分配该权限的子账号调受限接口 → 预期 **403**。
- [x] **校验失败**：发布活动不传 `slots`、报名传空 `slotIds` → 预期 **400** 带字段校验信息。
- [x] **白名单公开接口**免 token 可达：`/v/auth/login/wechat`、`/v/auth/login/dev`、`/v/auth/agreement`、`/v/auth/sms/codes`、`/a/auth/login`、`/a/auth/sms/codes`、`/a/auth/password/reset`。

## 附录 · 本版不实现 / 占位，勿测

> 测到这些返回空/占位/404 属正常，不是 bug（以 `url文档v1.md` 末尾「暂缓接口」为准）。

- **活动临时负责人考试**、**报名管理团队问卷审批**：V1 预留，本期不建。
