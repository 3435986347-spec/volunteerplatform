## 二、模块划分（按领域垂直切分）

后端采用**领域垂直切分的模块化单体**：父工程 `hengde-volunteer-parent`（packaging=pom，纯依赖管理）下**直接挂各个领域模块**，每个领域是一个 jar，内部自带 `controller / service / dao / entity / dto·vo` 三层；`hengde-volunteer-api` 持有唯一 `@SpringBootApplication` 启动类，依赖全部领域模块，是唯一可部署单元。

> 注：本方案取代了早期「按层聚合（dao 聚合 / service 聚合）」的拆分思路。

```
hengde-volunteer-parent              ← 父工程，仅做版本/依赖管理（packaging=pom）
├── hengde-volunteer-common          ← 公共工具/基础设施（统一返回体、全局异常、配置、工具类）
├── hengde-volunteer-auth            ← 账号/短信登录、忘记密码、志愿者注册、实名认证、企业微信群校验
├── hengde-volunteer-user            ← 基本信息、志愿者证、协会证书、i志愿证书、归属组织、我的保险、地址管理、安全中心
├── hengde-volunteer-activity        ← 志愿活动、活动相册、推荐活动、我的活动、服务记录、活动负责人
├── hengde-volunteer-organization    ← 组织架构、志愿小组、活动临时负责人考试、报名管理团队
├── hengde-volunteer-donate          ← 积分兑换、助学助困、圆梦微心愿、项目众筹、公益捐书、捐赠记录、捐书记录、结对中心、微心愿中心
├── hengde-volunteer-honor           ← 排行榜、榜样、评优评先、奖惩中心
├── hengde-volunteer-social          ← 社区(最热/关注/官方)、私信、互动、举报
├── hengde-volunteer-enterprise      ← 爱心企业
├── hengde-volunteer-publicity       ← 名单公示、公告栏、轮播图、文件下载
├── hengde-volunteer-data            ← 数据看板、积分中心、投诉中心
└── hengde-volunteer-api (jar)       ← 启动类 + 全局配置；依赖以上全部领域模块，聚合首页/社区/我的接口
```

每个领域模块的内部分包约定（以 user 为例）：

```
hengde-volunteer-user (jar)
└── com.hengde.user
    ├── controller   ← REST 接口
    ├── service      ← 业务逻辑（领域间互调走此接口）
    ├── dao          ← MyBatis-Plus Mapper
    ├── entity       ← 数据库实体
    └── dto / vo     ← 出入参（用 MapStruct 与 entity 互转）
```

## 三、POM 约定

- 公共坐标：`groupId = com.hengde`，`version = 1.0-SNAPSHOT`。
- **所有领域模块与 common 的 `<parent>` 都直接指向 `hengde-volunteer-parent`**（扁平结构，无中间聚合层）。
- 父工程 `packaging=pom`；其余模块 `packaging=jar`（含 common、各领域、api）。
- 子模块引依赖**不写 version**，版本统一由父 POM 的 `dependencyManagement` 锁定。
- 只有 `hengde-volunteer-api` 含启动类与 `main`，其余领域模块都是被依赖的库（不可独立启动）。
- Flyway 迁移脚本（`resources/db/migration/`）与 `application.yml` 等运行期配置集中放在 `api`（或 common），由 api 统一加载。
- 领域之间避免循环依赖；公共能力下沉到 `hengde-volunteer-common`。

典型领域模块 POM（以 user 为例）：

```xml
<parent>
    <groupId>com.hengde</groupId>
    <artifactId>hengde-volunteer-parent</artifactId>
    <version>1.0-SNAPSHOT</version>
</parent>
<artifactId>hengde-volunteer-user</artifactId>
<packaging>jar</packaging>
```

## 四、多系统部署与监管后台

本系统需支持**为多个社会组织各自部署一套一模一样的小程序 + 后台**（白标 / 多实例），对应需求见《小程序设想【第十版】.xlsx》末尾「监管后台」部分。

**部署模型：白标多实例。** 每个社会组织 = 一套独立部署：独立数据库、独立配置（小程序 AppId、微信/支付/短信密钥、协会名称与 LOGO 等），运行的是同一份 `hengde-volunteer-api` 制品。开新组织即新建一套配置 + 库并部署一个实例，互不影响、数据天然隔离。

**监管后台（运营方专用，独立于各组织实例之上）：**

- **多系统部署** — 一键为新社会组织开通并部署同款小程序及后台。
- **系统账号管理** — 管理每套部署的后台账号密码、有效期、权限。
- **增值功能收费** — 为每套系统按年分配短信、存储、身份证校验次数等额度，超额可追加购买。
- **数据管理** — 汇总监管各组织实例的数据与使用情况。
- **切换账号** — 直接切入任意组织后台。

> 监管后台是运营方的**平台级控制台**，与单个志愿者后台是不同的系统，建议作为独立应用，不放进 `hengde-volunteer-api` 的领域模块里。
>
> 说明：「多组织部署」靠的是白标多实例 + 外部化配置，**与单实例后端是单体还是微服务无关**。当前 `hengde-volunteer-api` 是模块化单体即可满足；是否引入 Spring Cloud 微服务取决于单实例自身的伸缩需求，而非部署多少个组织。
